## Задача

Реализовать умножение двух квадратных матриц размера от 100×100 до 2000×2000:

- последовательный вариант на CPU;
- параллельный вариант на GPU с использованием CUDA.

Для каждого размера матрицы измеряются времена выполнения на CPU и GPU и считается ускорение.

---

## Реализация
Матрицы хранятся построчно, элемент `C[i, j]` вычисляется как `C[i, j] = sum(A[i, k] * B[k, j])`
- **CPU** (`matmul_cpu`) — тройной цикл, сложность O(M·K·N), без оптимизаций.
- **NumPy** (`matmul_numpy`) — A @ B, использует оптимизированные BLAS-библиотеки. Это быстрый вариант на CPU.
- **GPU** (`matmul_gpu`) - используется библиотека CuPy, которая автоматические распраллеливает процессы с помощью `cuBLAS`. Итоговая матрица делится на `tiles`.
  Каждый тайл обрабатывается отдельным `thread block`.
  - копирование данных на GPU: `cp.asarray`;
  - умножение через `@` (`cuBLAS`);
  - копирование результата обратно: `cp.asnumpy`.

### Выводы

- **CPU** — простая реализация, однако очень медленная. Если в случае малых размерностей матриц это терпимо, то при больших размерностях становится непригодно для использования.  
- **NumPy** — быстрая реализация с помощью **CPU**. Примечательно, что при малых размерх матриц работает быстрее, чем GPU-реализация, так как время, затрачиваемое на пересылку данных между **CPU** и **GPU** больше самого времени вычисления.  
- **GPU** — быстрее на больших размерностях матриц, и чем она больше, тем больше ускорение относительно **CPU** и **NumPy** и **CPU** вариантов.
